## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge inner padding |
| sm | 8px | Gap between bench chips (gap-2), badge margin |
| md | 16px | Default padding (p-4), element spacing |
| lg | 24px | Section spacing (space-y-6) |
| xl | 32px | Layout gaps between courts |
| 2xl | 48px | Bottom bar clearance (pb-48 on scroll content) |
| 3xl | 64px | Not used in this phase |

Exceptions:
- Touch targets: chip min-height is 44px (`min-h-[44px]`) — tap-target floor, not a spacing token
- Add Court button: 44px minimum touch target (circle-plus affordance, inline between last court and bench)
- Bottom bar: `.fixed-safe-bottom` = `bottom: calc(5rem + env(safe-area-inset-bottom, 0px))` — device-specific, not overridden

Source: Inherited from Phase 13 UI-SPEC. No new exceptions beyond Add Court touch target.

## Sit-Out Count Badge Contract

Source: CONTEXT.md D-06.

| Property | Value |
|----------|-------|
| Applies to | Bench chips only (`[data-zone="bench"]` chips) |
| Data source | `session.rounds` iteration — count of rounds where each player ID appears in `round.sittingOut`; current draft NOT included |
| Display | A small gray pill/label below the player name on the chip |
| Badge copy | `"{N}×"` — e.g., "2×" for a player who has sat out twice |
| Badge style | `text-xs font-medium text-gray-500` — displayed as a line below the player name inside the chip |
| Court chips | Do NOT display sit-out badges |
| Zero sit-outs | Show "0×" — explicit is clearer than absent |

Chip HTML structure (bench only):
```
<div data-player-id="{id}" class="px-3 py-2 border rounded-full text-sm font-medium text-center min-h-[44px] flex flex-col items-center justify-center cursor-grab">
  <span>{playerName}</span>
  <span class="text-xs font-medium text-gray-500">{count}×</span>
</div>
```

Source for sit-out count calculation: `RoundDisplay.js` lines ~58-63 — `session.rounds.forEach(round => round.sittingOut.forEach(id => sitCounts[id] = (sitCounts[id] || 0) + 1))`.

## Toast Contract

Source: CONTEXT.md D-05 (Easter Egg thresholds).

No existing toast pattern in the codebase — implement inline as an ephemeral div.

| Property | Value |
|----------|-------|
| Position | Fixed, top of viewport, centered — `fixed top-4 left-0 right-0 flex justify-center z-50` |
| Container | `bg-gray-900 text-white rounded-xl px-4 py-3 max-w-xs mx-auto text-sm font-medium shadow-lg` |
| Duration | 2.5 seconds — then auto-remove from DOM |
| Animation | Fade in via `bounceIn` keyframe (already defined in `src/style.css`) on appear; fade out via 0.2s opacity transition on remove |
| Threshold 1 copy | "Oooh, more than Wimbledon's Championship courts? Fancy" — fires when court #20 is added (count crosses 19); court IS added |
| Threshold 2 copy | "Can't be better than Wimbledon!" — fires when at 55 courts and Add is tapped; court is NOT added |
| Implementation | Single `showToast(message)` function; appends div, sets timeout, removes |

## Haptic Feedback Contract

Source: CONTEXT.md D-07, `src/services/haptics.js`.

| Event | Haptic |
|-------|--------|
| Successful chip drop (any valid target) | `Haptics.medium()` — 20ms vibration |
| Trigger point | Inside `handleDragEnd(evt)`, after `reconcileDraftFromDOM` + `syncEmptySlots` + `validateAndUpdateUI` |

"Successful" means `onEnd` fired without the move being rejected by `onMove` (i.e., the chip landed in a valid zone). Call `Haptics.medium()` unconditionally in `handleDragEnd` — the SortableJS `onMove` returning `false` prevents `onEnd` from firing at all, so `onEnd` always implies success.

Import pattern: `import { Haptics } from '../services/haptics.js'` — already used in `RoundDisplay.js`, same pattern for `MatchEditor.js`.

## Validation Contract (Unchanged from Phase 13)

| Rule | Visual Response |
|------|----------------|
| Court has exactly 1 player | Red border `border-red-400`; inline label "needs 2+ players" in `text-xs font-bold text-red-600` |
| Court has 0 players | No error (empty courts are pruned on save, not flagged as errors) |
| Court has 2, 3, or 4 players | No error indicator |
| Any invalid court | Confirm button disabled: `opacity-50 cursor-not-allowed bg-gray-300 text-gray-500` |
| All courts valid | Confirm button active: `bg-blue-600 text-white shadow-lg shadow-blue-200` |

Validation fires reactively on every `handleDragEnd` event AND after every add/remove court action.

## Save / Confirm Contract (Phase 14 Addition)

Source: CONTEXT.md D-03.

On Confirm tap (before calling `SessionService.updateRound`):
1. Filter `_draft.courts` to remove any court where `teamA.length === 0 && teamB.length === 0`
2. Pass the pruned round to `SessionService.updateRound(_roundIndex, _draft)`
3. Navigate to `/active`

Pruning is silent — no toast, no confirmation dialog. Empty courts are simply excluded from the saved round.

# Phase 15 — UI Design Contract: Dark Mode Foundation

> Visual and interaction contract for Phase 15. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Note

Phase 15 is a **foundation-only phase** — no new views, no new interactive elements, no per-component dark overrides. The UI surface this phase touches is exactly:

1. `<html>` element — receives/loses `dark` class
2. `<body>` element — gets two additional Tailwind classes for baseline dark background
3. `src/style.css` — gains one `@custom-variant` line
4. `index.html` — gains one inline `<script>` block in `<head>`
5. `src/services/theme.js` — new file, no rendered output

Per-view dark overrides and the Settings toggle are Phase 16 scope. Do not add `dark:` utilities to any view in this phase.

## FOUC Prevention Script Contract

The inline script added to `<head>` in `index.html` must follow this exact behavior contract:

```
Read localStorage.getItem('pb:theme')
  └─ value === 'dark'          → add class 'dark' to <html>
  └─ value === 'light'         → do nothing (default is light)
  └─ value === 'auto' or null  → check window.matchMedia('(prefers-color-scheme: dark)').matches
                                    └─ true  → add class 'dark' to <html>
                                    └─ false → do nothing
```

Placement: `<script>` tag (no `type="module"`) placed in `<head>` **before** any `<link>` or other `<script>` tags.

Key name used: `pb:theme` (plain string: `'light'|'dark'|'auto'` or absent/null).

## ThemeService API Contract

File: `src/services/theme.js`

| Method | Signature | Behavior |
|--------|-----------|----------|
| `ThemeService.init()` | `init() → void` | Reads stored mode, calls `applyTheme()`, attaches `matchMedia` change listener (only active when mode is `'auto'` or unset) |
| `ThemeService.setMode(mode)` | `setMode('auto'|'light'|'dark') → void` | Persists to `localStorage` at key `pb:theme`, calls `applyTheme()` |
| `ThemeService.getMode()` | `getMode() → 'auto'|'light'|'dark'` | Returns stored mode; defaults to `'auto'` if key is absent |
| `ThemeService.applyTheme()` | `applyTheme() → void` | Internal; reads mode + system pref, adds or removes class `dark` on `document.documentElement` |

Storage access: ThemeService reads/writes `localStorage.getItem('pb:theme')` / `localStorage.setItem('pb:theme', value)` directly (encapsulated — callers never access localStorage). This satisfies DARK-04: callers use `ThemeService.setMode()`; ThemeService is the single point of access.

Call site: `ThemeService.init()` called in `src/main.js` before the router mounts any view.

## CSS Variant Contract

Add exactly one line to `src/style.css`, after `@import "tailwindcss"`:

```css
@custom-variant dark (&:where(.dark, .dark *));
```

This enables `dark:` utilities throughout the app (used by Phase 16). Phase 15 only adds this declaration; Phase 16 applies `dark:` utilities to view components.

## Interaction States

Phase 15 has no interactive UI. The only "interaction" is the OS-level system preference change event:

| Trigger | Behavior |
|---------|---------|
| OS switches dark/light while app is open and mode is `'auto'` | `matchMedia` change listener fires; `ThemeService.applyTheme()` runs immediately; `dark` class added/removed on `<html>` with no page reload |
| OS switches while mode is `'light'` or `'dark'` | No reaction — system preference is ignored when manual override is set |
